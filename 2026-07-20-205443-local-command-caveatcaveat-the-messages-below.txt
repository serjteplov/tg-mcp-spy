  ūüĒī Correctness Gaps

  1. _normalize_identifier strips the - sign for negative numeric ids, diverging
  from spec R20

  spec.md R20: "Tools SHALL accept channel identifiers as either a Telegram 
  username or a numeric channel id (including -100...)."

  _normalize_identifier does int(cleaned) then if telegram_id < 0: telegram_id =
  -telegram_id. This converts -1001234 to 1001234, dropping the -100 prefix.
  That is correct for DB lookup (since telegram.py also normalizes the same way
  and the DB stores the positive telegram_id). However, the same function is
  used in _resolve_db_channel line 135 where the fallback calls
  app.client.resolve_identifier(identifier.strip()) ‚ÄĒ this passes the original 
  raw identifier, not the normalized one. That's actually fine because
  TelegramClientWrapper.resolve_identifier does its own normalization. But the
  dual-normalization path is fragile: if _normalize_identifier and
  TelegramClientWrapper.resolve_identifier ever diverge in how they parse
  -100..., the DB lookup and Telegram resolution will target different entities.
  This is a latent correctness risk, not a current bug.

  2. upsert_posts in db.py has no UniqueConstraint on (channel_id, 
  telegram_message_id)

  design.md specifies: "Unique constraint on (channel_id, telegram_message_id)."
  The current posts_table definition omits this constraint. Duplicate
  prevention relies solely on application-level filtering (existing set check in
  _SyncRepository.upsert_posts). Under concurrent access (currently prevented
  by sequential processing), this could allow duplicate rows. Deviation from the
  approved design.

  3. _update_channel calls purge_old_posts globally, not per-channel

  design.md says: "A background purge (triggered at the start of update_channel

  current implementation purges all channels' old posts at the start of every
  update_channel call. This is functionally correct but potentially surprising ‚ÄĒ
  calling update_channel("A") deletes old posts from channel B too. Not a spec
  violation (the spec doesn't restrict the purge scope), but worth noting as a
  design gap.
